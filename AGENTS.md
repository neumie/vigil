# AGENTS.md

Pipeline: poll -> solve -> dispatch.

## Self-maintenance

When you discover an undocumented gotcha, abstraction, mandatory pattern, or a subsystem missing from this file, **update AGENTS.md before finishing the task** (`CLAUDE.md` is a symlink to it). Stale instructions cost the next agent more than they cost you to fix now.

## Mandatory abstractions

Core unions / interfaces gate where code goes. Bypass them and the daemon silently desyncs.

- **`TaskProvider`** (`src/providers/provider.ts`) — every external task source goes through this. Never import `contember.ts` (or any provider impl) outside `src/providers/`.
  - Implement four methods: `pollNewTasks`, `getTaskContext`, `resolveTaskSummary`, `postComment`. `resolveTaskSummary` seeds a source-backed solve Item when `POST /api/items/source` is invoked.
  - Extend the discriminated union in `src/config.ts`, register in `src/providers/registry.ts`.
- **`ItemCommands` + `ItemStore`** (`src/items/commands.ts`, `src/items/store.ts`) — every Item lifecycle write goes through commands; every Item persistence/event query goes through the store. Provider-discovered Items carry `Source` and start `unverified`; source-less hand-added Items start `queued`. Approve/reject transitions must go through `approveItem` / `rejectItem` so `queuedAt` / `completedAt` and `item_events` stay in sync. Do not write Item status or `item_events` directly from routes, poller, UI, or queue code.
  - Active-run transitions are guarded: `completeSolveItem`, `completeLoopItem`, `failItem`, and `cancelProcessingItem` only apply to `processing` Items. Use `startItem` first in tests/fixtures that model a completed run; use `cancelQueuedItem` for queued/planned cancellation.
  - Generic Item event writes are guarded for run-scoped event families: `recordEvent(..., 'solve_*')` only applies to `processing` solve Items, and `recordEvent(..., 'dispatch_failed')` only applies to `review` solve Items.
  - `recordEvent` is not an escape hatch for lifecycle events. Reserved event names with dedicated command methods (`item_*`, `solve_completed`, `loop_completed`, `almanac_run_started`, `plan_prepared`, `pr_created`, `comment_posted`, `dispatch_skipped`, `action_completed`) must go through those methods so status/timestamps and events cannot diverge.
  - Item planning identity and the `plan_prepared` event go through `recordPlanPrepared`. Routes may preflight status before spawning a planning surface, but the final workspace identity/event write still belongs to `ItemCommands`.
  - Execution workspace identity (`recordExecutionWorkspaceIdentity`) is processing-only. Call `startItem` first on solve/loop execution paths; use `recordPlanPrepared` for pre-run planned worktrees in routes/tests/fixtures.
  - Dashboard/API Plan intent is the exception to the source-less default: `POST /api/items` with `intent: "plan"` creates `planned` Items with no `queuedAt` and does not wake the Drainer. `planned` Items are startable/cancellable through the same Item routes; Drainer only auto-pulls `queued` Items.
  - Fan-out goes through `createSolveItems` / `createRalphItems` / `createHardenItems` with `parallelism`; those methods create one shared `GroupId` only when `parallelism > 1`. Do not set `groupId` from routes/UI or create sibling rows by looping outside `ItemCommands`.
- **Item workspace identity** (`src/items/identity.ts`) — Item-scoped worktree inputs resolve through `resolveItemWorkspace(item)`, which returns captured `baseRef`, stable `planDirName`, stable `branchName`, and an existing worktree path only if it still exists. Do not derive Item branches from mutable project config.
  - Forked Items may carry another Item's `branchName` as `baseRef` via `baseItemId`. Worktree creation resolves BaseRef through `resolveWorktreeStartPoint()` so remote branches, local Item branches, and SHAs work; do not hard-prefix `origin/` outside `src/worktree/manager.ts`.
  - **AI branch naming (`src/items/naming.ts`)** — `ensureItemWorkspaceName(...)` optionally replaces the default `vigil/item/<slug>-<suffix>` branch with a conventional model-derived name (`feat/…`, `fix/…`) when `solver.nameModel.enabled`. Gated entirely by that flag (default off) and scoped to **solve Items only — enforced inside `ensureItemWorkspaceName` (`if (item.kind !== 'solve') return item`)**, so neither `processSolveItem` (`src/queue/worker.ts`) nor the Item plan route (`src/server/routes/api.ts`) can name a ralph/harden Item; loop Items keep the deterministic name (their title is a PRD path / harden target). It runs **before** `resolveItemWorkspace` and persists through `ItemCommands.recordDerivedWorkspaceName`. It is fail-safe: any disablement, model failure, timeout, or unparseable output returns the unchanged Item so the deterministic `vigil/item/` default applies. It re-throws only cancellation (an aborted `signal`) — worker call sites run inside the pipeline's abort-aware catch; the plan route passes `c.req.raw.signal` and maps the abort to 503 (`isCancellation(err, c.req.raw.signal)`). `parseBranchName` scans stdout **bottom-up** and returns the **last** `<type>/<slug>` token (positional, across match shapes — a later labeled/trailing answer beats an earlier clean preamble line), tolerating trailing text on the line. The slug is clamped to a ~50-char budget. **Uniqueness is reserved atomically inside `recordDerivedWorkspaceName`**: the DB existence check, suffix decision, and write run synchronously (better-sqlite3, no await between), so two concurrent solves can't both reserve the same `base`; the git check (`localBranchExists` + `remoteBranchExists` — `refs/remotes/origin/*` then `git ls-remote`) is folded in as `gitTaken`, and a collision appends `-<itemSuffix>`. The model call goes through `runOneShot` (`src/solver/one-shot.ts`), a one-shot print-mode wrapper over `spawn-claude.ts` — NOT the agentic `AgentAdapter.buildHeadlessInvocation()` envelope. It runs in a **throwaway temp dir** (not the repo) so an agentic CLI with broad permissions can't touch the canonical tree, and both agents pass their permission/sandbox bypass so a no-tool call can't stall on an approval prompt. Default timeout 30s. Model id defaults per agent (claude → `claude-haiku-4-5`, codex → `gpt-5-mini`) unless `solver.nameModel.model` overrides.
- **`Drainer`** (`src/queue/drainer.ts`) — lane-aware scheduler. It pulls queued solve Items and queued loop Items (ralph/harden) from `ItemStore` oldest-first (`queuedAt ASC` within each lane), enforces solve-lane capacity from `solver.concurrency`, and runs loop Items in a separate lane (`loopCapacity()` currently 1). On start it recovers stale `processing` Items back to `queued` through `ItemCommands` before scheduling. Item start/retry/cancel/pause/resume routes go through `Drainer`; Item status and events still go through `ItemCommands`. Don't reintroduce `TaskQueue` as the daemon scheduler.
- **`LoopRunner`** (`src/queue/loop-runner.ts`) — almanac loop execution seam. `processLoopItem` creates/reuses the Item worktree, then `AlmanacLoopRunner.runLoop()` shells out to `almanac ralph` or `almanac harden`, captures the emitted `AlmanacRunId`, and writes `.<kind>-stop` on cancellation (`.ralph-stop` / `.harden-stop`). Do not send loop Items through `Solver`, and do not spawn almanac directly from routes or `Drainer`.
  - `runLoop()` must flush its output log stream before resolving or rethrowing; tests and run observation may read `logs/<itemId>.log` immediately after the loop returns.
- **`Solver`** (`src/solver/solver.ts`) — every code-execution backend implements `solve(SolveParams) → SolveResult`. Solvers build the final prompt after worktree creation and must call `SolveParams.onWorktreeReady(worktreePath)` immediately after creating/reusing the worktree, then `SolveParams.onPromptSnapshot(prompt)` immediately after rendering the exact prompt and before invoking the agent. This keeps cancelled/failed Items inspectable and dashboard display tied to immutable solve input. Never spawn `claude` (or any agent CLI) outside a `Solver` impl; reuse `src/solver/invoker.ts` (which wraps `spawn-claude.ts`) instead of re-implementing process spawning. The one sanctioned exception is `runOneShot` (`src/solver/one-shot.ts`) for cheap non-agentic completions (AI branch naming) — it still goes through `spawn-claude.ts`, never a raw `spawn`, and never re-implements the spawn primitive. Constructed via **`createSolver(config)`** (`src/solver/registry.ts`) — the single construction site, mirroring `providers/registry.ts`. The configured `solver.type` is the active type — **no fallback**: if `okena` is configured it's used or fails loudly, never silently swapped for `DefaultSolver`. Optional/extension solvers still load via dynamic `import()` so a missing optional dep can't crash module load. To add a backend: implement `Solver`, extend the `solver.type` enum in `src/config.ts`, add a branch in `src/solver/registry.ts`. Never `new` a solver or `import()` an extension solver directly at a call site — go through `createSolver`.
- **`AgentAdapter`** (`src/solver/agent-adapter.ts`) — owns agent CLI variation inside Solver/Spawner code: headless command shape, interactive command string, display label, and timeline parsing for `claude` vs `codex`. Call `createAgentAdapter(config.solver)` or the compatibility exports from `src/solver/agent-command.ts`; do not add agent-specific branches in `DefaultSolver`, `invoker`, Okena adapters, or spawners.
- **`Spawner`** (`src/spawner/spawner.ts`) — every interactive planning launch goes through this. Solvers do not expose planning methods. `POST /api/items/:id/plan` calls the active Spawner, which creates/reuses the worktree, writes `context.md` / `.planning-prompt.txt` through `PlanWorkspace`, and returns a user-facing hint. Construct via **`createSpawner(config)`** (`src/spawner/registry.ts`); the default comes from `config.spawner.name`, not `solver.type`. Extension spawners (e.g. Okena) load via dynamic `import()` and are discovered from adapter files. To add a planning surface: implement `Spawner` under `src/extensions/<name>/spawner.ts`, export `createSpawner(config)`, and keep solve execution untouched. Do not extend a closed Spawner enum — `spawnerNameSchema` validates name syntax only, while availability comes from installed adapter files.
  - Planning responses include the actual `spawner` name. Dashboard and extension planning copy must name that Spawner, not `solver.type` or `solver.agent`; the solver agent only shapes the planning prompt.
  - Items may carry a stored `spawner` preference. `POST /api/items/:id/plan` resolves that preference through `createSpawner(config, name)` when it differs from the active default and records the actual spawner name in the `plan_prepared` event. Item creation accepts only installed adapters from `listSpawnerAdapters()`.
  - Item planning is rejected while an Item is `processing`; planning writes worktree prompt artifacts and must not mutate an active solve/loop run's workspace.
- **`RunObservation`** (`src/items/observation.ts`) — read module for live/completed Item run state. It normalizes Item events, `logs/<itemId>.log`, PR status, and loop `.almanac/runs/<AlmanacRunId>/status.tsv` into one dashboard shape, degrading missing files/CLI failures to `unknown`/empty fields. API Item routes must pass `observeItemRun(item, { store: db.items })` into the Dashboard Contract helper (`toDashboardItem` / `toDashboardItemWithSiblings`); keep `toDashboardItem` side-effect-free by using its default empty observation only when no IO-backed observation was requested.
- **Dashboard Item lists** (`src/items/contract.ts`) — use `toDashboardItems(items, observationFor)` for list responses so grouped siblings render together with server-owned `group.label` plus `group.position` / `group.size`; use `toDashboardItem` only for a single item read or a route that intentionally lacks sibling context. If a list route paginates before contract conversion, expand any visible `groupId` to all siblings first so pagination does not render partial groups.
  - Single Item API responses with DB access should use the sibling-aware contract path (`toDashboardItemWithSiblings` plus `ItemCommands.listGroupItems`) so a grouped Item keeps its server-owned group label outside the list route.
  - Item action routes (`approve` / `reject` / `start` / `cancel` / `retry`) return the updated `DashboardItem` contract, not ad hoc message bodies, so web and extension clients stay tied to server-owned status/action rules.
  - `DashboardItem.allowedActions` is a UI command contract, not a wishlist. Only emit actions backed by an Item route and `ItemCommands`/`Drainer` behavior (`approve`, `reject`, `start`, `cancel`, `retry` today). Do not advertise `skip` / `complete` or any future action until the route and lifecycle write path exist.
  - Web list tabs and browser-title attention counts share `partitionWorkEntries` / `workAttentionCounts` in `web/src/components/TaskList.tsx`; do not duplicate status filters in `App.tsx`, because `planned` and `unverified` Items must count as waiting work just like the queued tab.
- **Config Document** (`src/config-document.ts`) — owns dashboard-safe config shape, default Spawner config (`spawner.name`), installed Spawner adapter metadata (`spawnerAdapters`), secret redaction, Settings edit metadata, and redacted-secret preservation on save. `/api/config`, `/api/config/full`, and config updates must go through this module; do not rebuild sanitized config, Spawner choices, or editable Settings fields inside routes or UI. Settings renders the metadata it receives, so adding/removing config fields means updating `src/config.ts` plus Config Document metadata in the same slice.
  - Config reads may tolerate stale raw config by parsing through the canonical schema, but config saves are strict: `parseConfigUpdate()` rejects unknown/stale fields before Zod can strip them. Do not bypass this with direct `configSchema.parse()` in routes.

Task-context + prompt assembly lives in `src/task-context.ts` (`formatTaskContext`, `buildTaskContext`) and `src/solver/prompt-builder.ts` (instruction templates). These are plain functions, not a pluggable seam — there was a per-project `TaskTransformer` registry but it was never used and got removed. If genuinely-different prompt shapes per project ever materialize, reintroduce a seam then.

## Pipeline (`src/queue/worker.ts`)

Solve Items run the five-phase `processSolveItem` path, launched by `Drainer`. Don't reorder, don't skip.

1. **Poll** — context comes from the Item payload or, for Source-backed Items, `provider.getTaskContext(item.source.externalId)`, merged through `buildSolveItemTaskContext`.
2. **Worktree + Solve** — delegated to the active `Solver`. Workspace identity comes from `resolveItemWorkspace(item)`, passing captured `baseRef` as the solver base branch. Returns `{ worktreePath, branchName, outcome }` where `outcome: { events, exitCode, rawOutput? }` is solver-produced.
3. **Persist timeline** — write `outcome.events` to `item_events` through `ItemCommands`. Each `Solver` owns how it produces events (DefaultSolver asks the configured `AgentAdapter` to parse CLI output; okena returns `[]`). The worker does NOT parse solver stdout.
4. **Parse result** — `new PlanWorkspace(worktreePath, planDirName).readResult()` reads `docs/plans/<planDirName>/solver-result.json`. No fallback: a missing/invalid file fails the Item. Shape is one Zod source (`src/solver/result-schema.ts`); keep the JSON template in `prompt-builder.ts` in sync with it.
5. **Dispatch** — `dispatchSolveItem()` records the PR if the agent pre-shipped one (`result.prUrl`); otherwise pushes the branch and opens a PR (`github.createPrs`), optionally posting a provider comment (`github.postComments`). Status/result/dispatch writes go to `items`/`item_events` through `ItemCommands`, so pre-shipped PRs, fallback PR creation, and provider comments stay behind Dispatch instead of Drainer. No tiering — the agent just ships.

Loop Items (ralph/harden) use `processLoopItem`, also launched by `Drainer`. It resolves Item workspace identity, reuses a planned worktree when present, otherwise creates one from captured `baseRef`, then calls `LoopRunner.runLoop()`. The runner executes `almanac ralph` with noninteractive flags from the Item payload/config or `almanac harden <target> --loop [--rounds N]`, captures the first emitted run id into `items.almanac_run_id`, writes the kind-specific stop file on cancellation, and reports completion/failure/cancellation through `ItemCommands`.

## Subsystems

- **`src/cli/`** — `vigil` binary wrapping launchd and scriptable Item creation: `start` / `stop` / `status` / `logs` / `add`. **`vigil add <solve|ralph|harden>` creates queued, source-less Items through `ItemCommands`** (including `parallelism`, BaseRef/BaseItem, and installed Spawner validation).
- **`src/extensions/okena/`** — alternative `Solver` and `Spawner` (local Okena daemon instead of raw CLI). Loaded via dynamic `import()` inside `createSolver` / `createSpawner`. Don't hard-import it. No `DefaultSolver` fallback — when `solver.type=okena`, okena is the active Solver even if momentarily unreachable; set `spawner.name=okena` separately to make Okena the default planning surface.
- **Planning artifacts (`docs/plans/<planDirName>/` inside the worktree)** — interactive plan phase happens before autonomous execution. `POST /api/items/:id/plan` calls `spawner.startPlanningSession(...)` (one call — ensures worktree, writes `context.md`, spawns/stages the planning prompt: okena runs the agent in a reused `plan: <title>` terminal; default stages `docs/plans/<planDirName>/.planning-prompt.txt` for the user to run). Persists `worktreePath`/`branchName`/`planDirName` on the Item row and writes `docs/plans/<planDirName>/README.md`. The user talks to the planning agent, runs `/grill-me <planDirName>` / `/grill-plan <planDirName>` / `/prd-create` (almanac) — those write `docs/plans/<planDirName>/brief.md`, `prd.md`, etc. The autonomous run reuses the same worktree; `buildTaskContext` (`src/task-context.ts`) calls `PlanWorkspace.readArtifacts()`, which reads every `*.md` in that dir, sorts by mtime, and wraps each in a `<plan_artifact>` block prepended before the task context. The on-disk layout (paths + IO for `context.md` / `.planning-prompt.txt` / `solver-result.json` / `README.md`) is owned by `PlanWorkspace` (`src/plan/workspace.ts`) — go through it, don't re-derive the path. Item planning resolves identity through `resolveItemWorkspace(item)`. Source-backed solve Item planning must fetch `provider.getTaskContext(item.source.externalId)` and merge it through `buildItemTaskContext(item, sourceContext)` before Spawner invocation; do not plan from the stored title/prompt summary alone.

### Directory rules

- **`extension/`** — **SolidJS, NOT React** (unlike `web/`). Use signals + `For`/`Show`, `on:click` not `onClick`, `class` not `className`, no hooks/`useState`. Host permissions: `http://localhost:*/*` only — never add tunnel/public URLs (manifest is canonical: `extension/manifest.json`).
  - `extension/package.json` is `type: commonjs`; Node tests that import extension TS through `tsx` must default-import the module and destructure exports instead of using named ESM imports.
  - Extension reloads can leave old content-script/popup code alive long enough for `chrome.storage` to throw `Extension context invalidated.`. Use `extension/src/storage.ts` (`getSync` / `setSync`) for storage access; do not call `chrome.storage.*` directly from extension code.
  - Untracked external tasks created from the extension go through `POST /api/items/source` (`api.createItemFromSource`) and become Source-backed, unverified solve Items via `ItemCommands`. The extension is Item-only — discover via `findItemBySource`, create via `createItemFromSource`, plan via `planItem`, act via `itemAction`.
  - `/api/items/source` accepts only `externalId`; source discovery still creates an unverified solve Item without selecting an agent. Item action routes that can launch solve work (`approve`, `start`, `retry`) may accept `solverAgent`; persist it through `ItemCommands.setSolveItemAgent` in the solve payload before waking the Drainer. `processSolveItem` must use that stored agent over `config.solver.agent`. Item planning route may also accept `solverAgent` because it shapes the planning prompt.
- **`web/`** — React 19 only. Never downgrade; code uses 19-only features (Actions, `use`, ref-as-prop).

## Gotchas

- **Worktree cwd.** Solver code runs with the worktree as `cwd`. Reading `process.cwd()` from anywhere outside `Solver` impls or the worker hits the wrong tree (or the daemon's tree). Prefer paths derived from `worktreePath` / `projectConfig.repoPath`. Exception: module-load-time daemon paths (config, DB, logs) capture the daemon's startup cwd — that's intended. Note: `src/server/routes/api.ts:291` resolves `process.cwd()` per request to read the log path — that works only because the daemon never `chdir`s, so don't copy the pattern into code that might run after a `cwd` change.
- **Optional solver/spawner imports.** Extension solvers/spawners (e.g. `src/extensions/okena/`) load via dynamic `import()` inside `createSolver` (`src/solver/registry.ts`) / `createSpawner` (`src/spawner/registry.ts`) — never a top-level static import from core code, so an unavailable optional dep can't crash module load. Add new solver backends as branches inside the solver registry; add new spawner backends as `src/extensions/<name>/spawner.ts` files exporting `createSpawner(config)` so the spawner registry discovers them.
- **Claude can pre-ship.** If `docs/plans/<planDirName>/solver-result.json` carries `prUrl`, `dispatcher.ts` records it and returns early (Claude shipped via `/almanac:ship`) — the push-branch-and-open-PR fallback only runs when `prUrl` is absent. Don't add side effects assuming the fallback always runs. For solve Items, `dispatchSolveItem()` posts provider comments only when `github.postComments` is enabled and `item.source.provider` matches the active provider name.
- **Okena token rotation.** Okena rotates the CLI token; `OkenaClient` reloads `cli.json` per call. Never cache the token in memory — long-running daemons will start returning HTTP 401 silently. Newly registered tokens activate only after Okena itself restarts.
- **Okena profile layout.** Okena keeps `cli.json`/`remote.json` under `profiles/<id>/`. `OkenaClient.resolveConfigDir()` reads `profiles.json` (`last_used` → `default_profile`) per call to find the active profile. Symptom of okena changing this layout again: `Error: Okena not configured` even though okena is running — re-check `resolveConfigDir()` against the new on-disk layout.
- **No solver fallback.** `createSolver` (`src/solver/registry.ts`) never substitutes a different solver: `solver.type=okena` → `OkenaSolver`, always. If okena is unreachable, the factory logs a warning but still returns `OkenaSolver`, so okena errors surface per task (logs + dashboard) instead of being masked, and recover on their own once okena is back (the client reloads its token per call — no daemon restart needed). `index.ts` logs configured + active via `solver.constructor.name`. Planning is configured independently through `spawner.name`; setting `solver.type=okena` alone no longer chooses Okena planning.
- **Pipeline error protocol.** How solve/loop-path code signals *which phase* failed and how cancellation is recognised lives in one module: `src/util/errors.ts`. Throw `phaseError(phase, msg)` (phase typed as `ErrorPhase` — a typo'd phase is a compile error, not a silent fall-through to the worker's default) and `taskCancelled()` (sets `name: 'AbortError'`). The worker's catch block classifies via `isCancellation(err, signal)` and `errorPhase(err)` (defaults to `'solve'` for an untagged throw). Use `phase: 'loop'` for almanac loop execution failures. The constructors and the classifiers are two halves of one interface — never hand-build `Object.assign(new Error(...), { phase })` or `{ name: 'AbortError' }` again, or the typed seam leaks. A throw with no `phase` is recorded as `'solve'`.
- **Solver result: one source, one read path.** Phase 4 reads `docs/plans/<planDirName>/solver-result.json` via `PlanWorkspace.readResult()` only — there is NO stdout fallback (deleted; okena produces no stdout anyway). A missing/invalid file is a hard failure (`phase: 'solve'`). The shape is a single Zod source in `src/solver/result-schema.ts` (`SolverResult = z.infer<typeof solverResultSchema>`, re-exported via `types.ts`); `z.object` strips unknown keys, so any field the agent writes (e.g. `prUrl`) MUST be declared there or it's silently dropped — and the JSON template in `prompt-builder.ts` must stay in sync with the schema. The result *path* is owned by `PlanWorkspace` (`src/plan/workspace.ts`): `prompt-builder.ts` references it via `planPaths()`, so the prompt, the reader, and the okena poll path can't disagree.
- **Stale processing recovery.** `Drainer.start()` recovers stale `processing` Items back to `queued` through `ItemCommands.recoverStaleProcessingItems()` and records `item_recovered` before lane scheduling. Don't write code that assumes Item `processing` always means currently running — it might be recovered from a prior crash.
- **Dev server startup has side effects before port bind.** `npm run dev` loads config, initializes DB (runs migrations), starts poller/drainer, then binds `server.port`. If an existing daemon already owns `:7474`, startup can still re-enqueue/poll before failing with `EADDRINUSE`. Check `lsof -iTCP:7474 -sTCP:LISTEN -n -P` or stop the daemon before smoke-running dev.
- **Legacy Task model is gone.** Migration 11 drops the `tasks` and `event_log` tables (migration 10 already dropped `tasks.tier`/`solver_confidence`/`chat_sessions`/`chat_messages`). Items are the only work model: `items` + `item_events` via `ItemStore`/`ItemCommands`. Do not re-add a `tasks` table, `TaskRecord`, `processTask`, `/api/tasks/*` routes, `resolveTaskWorkspace`, `vigil run`, or `event_log`. The `TaskContext`/`TaskProvider`/`TaskSummary` seam (`src/providers/provider.ts`) is NOT legacy — it's the external-task representation that feeds Items; keep it. `poll_state` (the provider watermark) survives migration 11.
- **Solvers/Spawners assemble their own prompts.** `Solver.solve()` and `Spawner.startPlanningSession()` receive the raw `taskContext` (not a pre-built string or thunk) and call `buildPrompt` / `buildPlanningPrompt` themselves AFTER worktree creation — so `buildTaskContext` can read worktree-resident `docs/plans/<planDirName>/*.md`. Don't move prompt assembly back into `worker.ts` / `api.ts`: they'd build prompts before the worktree exists and miss plan artifacts.
- **Solve prompt snapshots are immutable.** Solvers must call `onPromptSnapshot(prompt)` after rendering the exact prompt and before agent invocation. Solve Items persist it to `items.solve_input_snapshot` through `ItemCommands.recordSolveInputSnapshot`. Workers and dashboard code must read that stored snapshot — never rebuild a prompt after solve completion, because plan artifacts may have changed.
- **Solve outcome is solver-owned.** `solve()` returns `outcome: { events, exitCode, rawOutput? }` (`src/solver/solver.ts`); the worker persists it verbatim and never parses solver stdout. `DefaultSolver` gets timeline events from `AgentAdapter.parseTimeline(stdout)`; `OkenaSolver` returns `events: []` and no `rawOutput` (it runs the agent in its own terminal), so okena tasks have no granular dashboard timeline and a null `claudeRawOutput` — don't write worker/dashboard code that assumes those are populated.
- **Plan endpoint creates worktree, autonomous run reuses it.** `POST /api/items/:id/plan` runs `Spawner.startPlanningSession`, persists `worktreePath`/`branchName`/`planDirName`. The worker reads them back and passes `existingWorktreePath` to `solve()` / loop execution. Solvers MUST honor `existingWorktreePath` (skip worktree creation) — otherwise the autonomous run wipes planning state the user wrote in `docs/plans/<planDirName>/`. For okena: solve always spawns a fresh terminal (never the user's `plan:` terminal); OkenaSpawner reuses an existing `plan:` terminal so repeated Plan clicks don't pile up windows.
- **Plan identity dates are runtime dates.** `resolveItemWorkspace()` derives default `planDirName` from `item.createdAt` (`YYYY-MM-DD-...`). Tests must derive the expected date from the Item row, not hard-code the date from an old RALPH iteration.

## Build & run

- **Ports.** Backend `:7474`, frontend dev server `:7475` proxying `/api` to backend.
- **Verification.** Module tests run through Node's built-in test runner via `npm run test` (`tests/**/*.test.ts`, loaded with `tsx`). Standard pre-commit loop is `npm run lint`, `make test`, `make check`; `make check` runs backend + web builds. If touching `extension/`, also run `node build.mjs` from `extension/` — the Makefile does not build the Solid extension. For daemon/pipeline changes, also run `npm run dev` and exercise the affected path end-to-end (poll → solve → dispatch, or whatever phase you touched).
- **Config.** Loads from `VIGIL_CONFIG` env or `./vigil.config.json`. **`src/config.ts` is the canonical schema** — read it instead of duplicating fields here.
- **Runtime dep: almanac.** The solver prompt (`src/solver/prompt-builder.ts`) references `/almanac:task-start`, `/almanac:branch-name`, `/almanac:ship`, `/almanac:commit`, and loop Items shell out to the `almanac ralph` / `almanac harden` CLI. These resolve via the user's globally-installed `almanac` plugin/CLI — vigil does NOT vendor or install its own copies. If the daemon host is missing almanac, solve agents may silently fail to load those skills and loop Items fail in `phase: 'loop'`. Install via almanac's own setup (`bash install.sh && almanac install claude-code` in the almanac repo).
- **Loop runtime artifacts.** `almanac ralph` / ralph-loop and `almanac harden` may write `.almanac/` run registry files and `docs/plans/<name>/*.log` / harden plan files. These are runtime artifacts and are gitignored; commit PRD/prompt/issue/rubric files when they are intentional, not run logs.
